# GMS2 MCP Server

MCP сервер для работы с проектами GameMaker Studio 2 в Cursor IDE и Claude.

## Что это такое?

Этот MCP сервер парсит и извлекает информацию из проектов GameMaker Studio 2, предоставляя разработчикам и AI-агентам быстрый доступ к структуре проекта, GML-коду и метаданным ассетов — без открытия GameMaker Studio 2.

**Основные возможности:**
- Для разработчиков: экспорт всех данных проекта в читаемом формате для изучения и анализа
- Для AI-агентов: быстрое понимание структуры проекта, значительно ускоряющее vibe-кодинг
- Глубокий анализ: автоматическое сканирование объектов, скриптов, комнат, спрайтов и их связей
- Поддержка записи: редактирование GML-файлов напрямую через AI-агента с автоматическим созданием резервных копий

## Структура проекта

```
gms2-mcp-server/
├── mcp-serv/
│   ├── mcp_server.py       # MCP сервер с 12 инструментами
│   └── gms2_parser.py      # Парсер проектов GameMaker Studio 2
├── docs/

│   └── README_RU.md        # Документация на русском

Дополнительно пользователь создает:
└── .cursor/mcp.json        # Конфигурация Cursor IDE (включает путь к проекту)
```

## Установка

### 1. Клонирование репозитория

```bash
git clone https://github.com/Atennebris/gms2-mcp-server
cd gms2-mcp-server
```

### 2. Создание виртуального окружения

```bash
# Создание venv
python -m venv venv

# Активация (Windows)
venv\Scripts\activate

# Активация (Linux/Mac)
source venv/bin/activate
```

### 3. Установка зависимостей

```bash
pip install -r requirements.txt
```

**Текущие зависимости:**
- `mcp==1.11.0` — официальный Python SDK для Model Context Protocol
- `python-dotenv==1.1.1` — загрузка конфигурации из .env файлов

### 4. Конфигурация Cursor IDE

Создайте файл `.cursor/mcp.json` в корне вашего проекта:

```json
{
  "mcpServers": {
    "gms2-mcp": {
      "command": "python",
      "args": ["C:/Users/YourName/Desktop/gms2-mcp-server/mcp-serv/mcp_server.py"],
      "env": {
        "GMS2_PROJECT_PATH": "C:/Users/YourName/Downloads/Your GMS2 Project"
      }
    }
  }
}
```

**Важно:**
- Замените путь в `args` на абсолютный путь к файлу `mcp-serv/mcp_server.py`.
- Замените путь в `env.GMS2_PROJECT_PATH` на абсолютный путь к папке, содержащей `.yyp` файл.
- Используйте прямые слеши `/` даже в Windows.

**Альтернатива — глобальная конфигурация (для всех проектов):**
1. В Cursor IDE откройте настройки.
2. Перейдите в раздел **Tools & Integrations**.
3. Нажмите **New MCP Server** внизу страницы.
4. Откроется глобальный `mcp.json` — добавьте туда ту же конфигурацию.

**Последовательность запуска:**
1. Cursor IDE запускает `python mcp-serv/mcp_server.py`.
2. `mcp_server.py` читает `GMS2_PROJECT_PATH` из переменной окружения (заданной в `mcp.json`). Также поддерживается файл `config.env` в той же директории, что и сервер.
3. `gms2_parser.py` выполняет всё парсирование. Экземпляр парсера кешируется на сессию — он не создаётся заново при каждом вызове.

### 5. Перезапуск Cursor IDE

После сохранения конфигурации перезапустите Cursor IDE и убедитесь, что MCP сервер отображается с зелёным статусом.

## Конфигурация

Сервер разрешает путь к проекту в следующем порядке приоритета:

1. Аргумент командной строки `--project-path` (для ручного тестирования).
2. Переменная окружения `GMS2_PROJECT_PATH` (задаётся в `mcp.json`).
3. Ключ `GMS2_PROJECT_PATH` в файле `mcp-serv/config.env` (опциональный запасной вариант).
4. Параметр `project_path`, переданный напрямую в вызов инструмента.

При обычном использовании с Cursor IDE достаточно только шага 2 — всё настраивается в `mcp.json`.

## Инструменты

После успешной настройки доступны **12 инструментов**:

---

### 1. `scan_gms2_project`

Сканирует директорию проекта GMS2 и возвращает полную структуру ассетов: объекты, скрипты, комнаты, спрайты, шрифты, звуки и другие. Включает количество GML-файлов для каждого ассета и список недавно изменённых GML-файлов.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `project_path` | Нет | Путь к папке проекта GMS2 (используется настроенный путь, если не задан) |

---

### 2. `get_gml_file_content`

Читает и возвращает полное содержимое конкретного GML-файла вместе с количеством строк и относительным путём.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `file_path` | Да | Путь к GML-файлу (относительный от корня проекта или абсолютный) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 3. `get_room_info`

Возвращает детальную информацию о комнате, разобранную из её `.yy` файла: слои, экземпляры объектов в комнате, настройки комнаты и размеры.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `room_name` | Да | Название комнаты (например, `rm_level1`) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 4. `get_object_info`

Возвращает метаданные объекта GMS2 из его `.yy` файла: назначенный спрайт, родительский объект, настройки физики и определения переменных.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `object_name` | Да | Название объекта (например, `obj_player`) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 5. `get_sprite_info`

Возвращает информацию о спрайте: путь, наличие `.yy` файла и список PNG-файлов кадров.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `sprite_name` | Да | Название спрайта (например, `spr_player`) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 6. `export_project_data`

Экспортирует все данные проекта — объекты, скрипты, комнаты, GML-код — в единый текстовый блок. Аналог функции экспорта из vibe2gml. Опционально может сохранить результат в файл.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `save_to_file` | Нет | Если `true`, записывает экспорт в файл вместо возврата в ответе (по умолчанию: `false`) |
| `output_file` | Нет | Путь к файлу для сохранения (используется только при `save_to_file: true`) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 7. `list_project_assets`

Выводит список всех ассетов проекта по категориям с указанием количества GML-файлов. Поддерживает фильтрацию по одной категории.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `category` | Нет | Фильтр по категории: `Objects`, `Scripts`, `Rooms`, `Sprites`, `Notes`, `Tile Sets`, `Timelines`, `Fonts`, `Sounds`, `Extensions` |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 8. `search_in_project`

Ищет текст или регулярное выражение во всех GML-файлах проекта. Возвращает имена файлов, номера строк и найденный текст.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `query` | Да | Текст или регулярное выражение для поиска |
| `case_sensitive` | Нет | Учитывать ли регистр (по умолчанию: `false`) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 9. `get_object_events`

Возвращает все события, определённые для объекта GMS2, вместе с полным GML-кодом каждого события. Позволяет понять поведение объекта за один вызов.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `object_name` | Да | Название объекта (например, `obj_enemy`) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 10. `get_script_content`

Читает GML-содержимое скрипта по имени, без необходимости знать путь к файлу.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `script_name` | Да | Название скрипта (например, `scr_utils`) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 11. `edit_gml_file`

Перезаписывает содержимое существующего GML-файла новым содержимым. Автоматически создаёт резервную копию `.bak` перед записью (если не отключено).

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `file_path` | Да | Путь к GML-файлу для редактирования (относительный или абсолютный) |
| `new_content` | Да | Новое GML-содержимое для записи |
| `create_backup` | Нет | Создать резервную копию `.bak` перед сохранением (по умолчанию: `true`) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

---

### 12. `find_asset_references`

Находит все упоминания ассета (объекта, скрипта, спрайта и т.д.) в GML-файлах и комнатах проекта. Возвращает имена файлов, номера строк и количество экземпляров в комнатах.

| Параметр | Обязателен | Описание |
|----------|------------|----------|
| `asset_name` | Да | Название ассета для поиска (например, `obj_player`) |
| `project_path` | Нет | Путь к папке проекта GMS2 |

## Безопасность

Сервер включает несколько защитных механизмов:

- **Валидация имён ассетов** — все параметры с именами ассетов проверяются по паттерну `[a-zA-Z0-9_-]`. Имена, содержащие разделители пути или другие специальные символы, отклоняются.
- **Ограничение доступа к файловой системе** — перед чтением или записью любого файла сервер разрешает реальный путь через `os.path.realpath` и проверяет, что он находится внутри директории проекта. Запросы, указывающие за пределы проекта, отклоняются с ошибкой доступа.
- **Изоляция ошибок** — внутренние исключения логируются на стороне сервера (в stderr через модуль `logging`) и не передаются в полном виде вызывающей стороне. Пользователь получает обобщённое сообщение об ошибке.

## Решение проблем

### MCP сервер показывает красный статус
1. Проверьте абсолютный путь в `args` внутри `.cursor/mcp.json`.
2. Убедитесь, что `GMS2_PROJECT_PATH` указывает на папку с `.yyp` файлом.
3. Убедитесь, что виртуальное окружение активировано и зависимости установлены.

### Сервер не находит проект
1. Проверьте, что путь в `GMS2_PROJECT_PATH` существует и содержит `.yyp` файл.
2. Используйте прямые слеши `/` даже в Windows.

### Инструменты не отображаются (0 tools)
1. Перезапустите Cursor IDE.
2. Убедитесь, что Python интерпретатор доступен из терминала.
3. Протестируйте сервер вручную: `python mcp-serv/mcp_server.py`

## Архитектурные замечания

Сервер построен на двух компонентах:

- **`mcp-serv/gms2_parser.py`** — разбирает `.yyp` и `.yy` файлы, сканирует GML-файлы, реализует всю логику получения и записи данных.
- **`mcp-serv/mcp_server.py`** — оборачивает парсер как MCP сервер. Отвечает за диспетчеризацию инструментов, валидацию входных данных, разрешение путей и централизованную обработку ошибок. Парсер создаётся один раз за сессию и кешируется.

Логирование осуществляется через стандартный модуль `logging` в stderr. По умолчанию отображаются только сообщения уровня `WARNING` и выше. Для подробного вывода при разработке установите уровень `DEBUG` в исходном коде.

## История проекта

Этот MCP сервер создан на основе идеи и функционала проекта [vibe2gml](https://github.com/zsturg/vibe2gml), который экспортирует проекты GMS2 в текстовый формат для работы с AI-агентами. Данный сервер развивает эту концепцию:

- Прямая интеграция с Cursor IDE и Claude через протокол MCP
- Расширенный набор инструментов (12 против исходного подхода только с экспортом)
- Доступ к данным в реальном времени без экспорта в файл
- Поддержка редактирования GML-файлов
